# Девлог

## Правила

- Це журнал сесій проєкту.
- Кожна сесія фіксує: проблему/ціль, що зроблено, складнощі, рішення, фідбек користувача.
- Якщо користувач дає новий важливий контекст пізніше, він може бути доданий у релевантний запис заднім числом.
- Журнал не повинен перетворюватися на changelog дрібниць; тут фіксуються рішення і рух по проєкту.

## 2026-04-03 - Сесія 001

### Проблема / Ціль

Зафіксувати керівний контур проєкту без змін коду:

- стратегічне бачення;
- короткостроковий план;
- журнал сесій для подальшої роботи.

### Зафіксований Контекст Користувача

- Проєкт починався як ранній GPT-3.5 чатбот.
- Згодом на нього нашаровувалися нові фічі, адаптації до нових моделей та зміни Telegram API.
- Кодова база накопичила багато історичних шарів, рудиментів і костилів.
- Бот зараз працює "плюс-мінус нормально", але архітектурно нестабільний.
- Бажана функціональність:
  - текстові відповіді;
  - voice replies на voice input;
  - озвучення тексту по `/a` та `/v`;
  - пам'ять по різних сесіях;
  - розуміння reply/mention "геометрії" Telegram;
  - image understanding;
  - у перспективі video understanding;
  - обробка тегнутого повідомлення разом із текстовою інструкцією користувача.
- Найболючіша продуктова проблема:
  - бот історично побудований навколо одного API-ключа;
  - потрібне розведення по різних ключах / моделях / провайдерах;
  - частина провайдерів не підтримує мультимодальність, тому потрібна capability-based маршрутизація.
- Довгостроково потрібен серверний "пульт керування" для ключів, моделей, промптів і режимів.
- Багатошарову пам'ять треба буде адаптувати з `D:\GODOT\chibigochi`, але це окрема задача.
- Старий код не є цінністю сам по собі; пріоритет лише в результаті.
- Поточний пошук у боті існує, але працює незадовільно:
  - це неконтрольований search behavior усередині логіки моделі;
  - результат часто повертається в машинному, сирому форматі;
  - пошук не оформлений як окремий керований pipeline.
- Цільовий підхід до пошуку:
  - молодша модель або planner вирішує, що саме треба підключити;
  - search є окремим інструментом;
  - сильніша або профільна модель формує фінальну відповідь;
  - за потреби search повторюється до трьох ітерацій.

### Що Зроблено

- Створено папку `docs/project/`.
- Створено `strategy.md` як стратегічну рамку перебудови бота.
- Створено `plan.md` як макроплан і короткостроковий план спринтів.
- Створено `devlog.md` як постійний журнал сесій.
- Зафіксовано принцип: не полірувати історичний хаос, а прагматично переписувати або видаляти те, що заважає.

### Складнощі

- На старті відсутня формалізована архітектурна рамка; контекст проєкту існує переважно в усному описі користувача та історичних нашаруваннях коду.
- Частина бажаної функціональності вже задумана, але не реалізована, тому важливо розводити:
  - що є поточним станом;
  - що є цільовою архітектурою.

### Рішення

- Першим артефактом стає не rewrite, а керівний контур планування.
- Найближчий практичний етап: формальний технічний аудит актуального репозиторію.
- Пам'ять з `chibigochi` не змішується з поточною стабілізацією; це окремий великий блок робіт.
- Пошук тепер зафіксований як окремий стратегічний напрям перебудови, а не як дрібне поліпшення.
- Цільовий search flow: planner -> search -> evaluator -> optional retry -> final response.

### Фідбек Користувача

- Не чіплятися за старий код як за цінність.
- Пріоритет: результат, навіть якщо для цього доведеться видалити великі шматки старої системи.
- Спочатку потрібна стратегія, потім короткі плани, потім покрокове виконання.

### Додаток Заднім Числом

Користувач додав важливе уточнення вже після створення початкових документів:

- пошук є реально використовуваною функцією;
- він повинен перейти з "чорної скриньки всередині моделі" у явний агентивний pipeline;
- фінальний текст після пошуку повинен бути людським і відповідати запиту користувача;
- треба підтримати до трьох search iterations, щоб уникнути слабких відповідей з першої спроби.

## 2026-04-03 - Сесія 002

### Проблема / Ціль

Додати окремий research-first шар перед технічним аудитом реалізації.

### Зафіксований Контекст Користувача

- Перед великим rewrite не можна покладатися на застарілі знання про моделі, API і агентні підходи.

- Користувач окремо рекомендував дивитися на напрацювання OpenClaw та Zed.
- Потрібно зібрати бібліотеку по напрямах, щоб далі не впиратися в незнання актуальних процесів.

### Що Зроблено

- Додано окрему research-бібліотеку `docs/research/`.
- Створено `README.md` як карту актуалізації знань.
- Створено `provider-landscape.md`.
- Створено `agentic-architectures.md`.
- Створено `multimodal-media.md`.
- Створено `search-stack.md`.
- Оновлено `strategy.md`, щоб research baseline став Phase 0.
- Оновлено `plan.md`, щоб research ішов перед технічним аудитом реалізації.

### Складнощі

- "Актуальні моделі" — рухома ціль, тому цей шар не може бути одноразовим документом.
- Потрібно було свідомо не з'їхати в нескінченний огляд усього ринку і тримати фокус на тому, що реально важливо для бота.

### Рішення

- Research baseline фіксується як окремий етап перед аудитом коду.
- OpenClaw використовується як архітектурний референс для control plane, failover, skills і routing, а не як щось, що треба копіювати цілком.
- Zed використовується як референс для процесної дисципліни, protocol-oriented integration і agent workflow hygiene.
- Перший research layer не є фінальним вибором стеку; це орієнтир для наступних рішень.

### Фідбек Користувача

- Не спиратися на застарілі знання.
- Спочатку дослідити сучасний landscape, потім вже будувати нову архітектуру.

## 2026-04-03 - Сесія 003

### Проблема / Ціль

Провести формальний технічний аудит актуального репозиторію крізь уже зібрану research-рамку.

### Зафіксований Контекст Користувача

- Користувач надав актуальний локальний `.env` для тестового/робочого контуру старого бота.
- З цього стало видно, що його поточний робочий конфіг узгоджується переважно з legacy-лінією, а не з новим runtime.
- Секрети не фіксувалися в документації або git.

### Що Зроблено

- Проведено архітектурний аудит актуального репозиторію.
- Зафіксовано, що в проєкті співіснують дві різні системи:
  - legacy `main.py` + `src/*`;
  - new runtime `run.py` + `adapters/app/agent/media/memory/db`.
- Окремо зіставлено env-контракти legacy і new runtime.
- Виявлено і задокументовано критичні поломки в:
  - `agent/runner.py`;
  - `media/router.py`;
  - `media/video.py`;
  - `src/message_handler.py`.
- Окремо зафіксовано, що deploy artifacts досі ведуть у legacy runtime, а tests/CI — у new runtime.
- Створено окремий audit-артефакт: `docs/project/audit-baseline-2026-04-03.md`.

### Складнощі

- Репозиторій містить явні merge-корупції, тому довелося відокремлювати:
  - структурно правильну нову архітектуру;
  - її фактичну поточну працездатність;
  - legacy-гілку, яка все ще лишається deploy path.
- Частина дефектів не просто архітектурні, а буквальні runtime bugs.

### Рішення

- Канонічним кандидатом на подальшу перебудову визнано new runtime на `run.py`.
- Legacy-шар `main.py` + `src/*` треба трактувати як legacy, а не як рівноправну основу для майбутньої роботи.
- Наступний етап має бути stabilization sprint, а не додавання нових фіч.

### Фідбек Користувача

- Користувач дав робочий `.env`, щоб полегшити аудит і майбутню стабілізацію.
- Пріоритет залишається той самий: не зберігати старий код заради старого коду, а рухатись до результату.

## 2026-04-03 - Сесія 004

### Проблема / Ціль

Запустити стабілізаційний спринт навколо `run.py` і перевірити, що новий рантайм хоча б імпортується та проходить критичні smoke-сценарії до того, як чіпати більшу архітектурну перебудову.

### Зафіксований Контекст Користувача

- Користувач хоче, щоб саме остання GitHub-версія стала робочою базою.
- Є спільний VPS для майбутнього staging, але на ньому не можна зачіпати вже наявні проєкти.
- Сервер можна використовувати лише як ізольовану тестову ціль, а не як місце для перезапису `comixator`.

### Що Зроблено

- Нормалізовано набір залежностей у локальному `venv` і встановлено runtime- та test-залежності.
- Виправлено контракт залежності `openai`, щоб код працював з `openai 1.x`, на який поточний рантайм уже і був розрахований.
- Додано `core/env.py` і підв'язано новий рантайм до legacy env-імен там, де це потрібно.
- Перероблено зламані runtime-файли, що були в merge-корумпованому стані: `agent/llm.py`, `agent/runner.py`, `media/router.py`, `media/video.py`.
- Вирівняно deploy/runtime entrypoints у бік `run.py`: `Procfile`, `Dockerfile`, `docker-compose.yml`.
- Додано ліниві імпорти `bs4`, щоб імпорт не валився до фактичного використання парсингу.
- Додано сумісність з Python 3.9 у `run.py` через postponed annotations.
- Виправлено проблеми надійності, знайдені під час ручної перевірки:
  - `app/message_logic.py` тепер терпить відсутні PTB entity/text-поля і для пошуку згадок відступає до `UnifiedMessage.text`;
  - `media/router.py` тепер терпить відсутній `reply_text` і використовує no-op callback для помилок, якщо його немає;
  - `agent/runner.py` тепер додає assistant tool-call message назад в історію перед результатами інструментів, що потрібно для реальної сумісності tool-loop.

### Перевірка

- `py_compile` пройшов на стабілізованих runtime-файлах.
- Імпортний smoke-check пройшов для `run`, `agent.runner`, `media.router`, `media.video`, `app.message_logic`.
- Ручний harness пройшов критичні сценарії:
  - agent search/tool loop;
  - PTB auth flow;
  - PTB media mention flow;
  - video cleanup flow.

### Складнощі

- Повний `pytest` локально все ще блокувався інфраструктурою, а не import/runtime-синтаксисом:
  - тестам потрібен `MariaDB`;
  - локальний Docker daemon був непридатний у межах цієї сесії;
  - локальний сервіс `MariaDB` не був встановлений.
- Наданий VPS придатний як майбутній ізольований staging-box, але на ньому тоді був лише Python 3.12 без Docker і без MariaDB/MySQL.
- Оскільки цей VPS уже хостить інші проєкти, встановлення нових системних сервісів треба було трактувати як окремий свідомий крок.

### Рішення

- Не чіпати на спільному VPS існуючий деплой `comixator`.
- Розглядати VPS як опційний ізольований staging лише після того, як локальна стабілізація зайде достатньо далеко.
- Продовжувати тиснути локальну стабілізацію спочатку, бо кілька збоїв були ще чистими code/runtime-дефектами й виправлялися без серверного ризику.

### Фідбек Користувача

- Спільний VPS можна використовувати для тестів, якщо не зачіпати інші проєкти.
- Наявні деплої на цьому VPS мають залишатися ізольованими.

### Фідбек Користувача

- Користувач спочатку хоче окреме Smartest-середовище на VPS.
- Домен `smartest.klawa.top` і конфіг-інтерфейс під паролем заплановані як наступний шар, а не як перезапис чинних серверних проєктів.
