# Протокол: Поведінковий контракт бота

> **Цей документ — конституція бота. Що в ньому зафіксовано — мусить працювати. Що не в ньому — fair game для рефактору.**
>
> **Аудиторія:** будь-яка модель (Claude, Codex, інша) або людина, яка вносить зміни в код Smartest.
>
> **Імператив для моделі:** перш ніж писати код — прочитай цей документ, потім `behavior-audit.md`. Якщо твоя зміна може торкнутись поведінки, описаної там — спочатку онови acceptance-тест, потім код. Якщо тест не валиться після твоїх змін — або тест поганий, або твої зміни нічого не змінили.

---

## 1. Навіщо це існує

### 1.1 Корінна проблема

Smartest — це бот, який накопичив багато шарів: команди, маршрутизація, vision, voice, search, memory, billing. Кожен шар має юніт-тести (`tests/test_NNN_*.py`), і це прекрасно для **внутрішньої** логіки. Але є системна вада:

> **Можна пройти всі 330 юніт-тестів і при цьому зламати 10 продуктових сценаріїв.**

Чому? Бо юніт-тести перевіряють "функція X на вході Y віддає Z", а **користувач** живе у термінах "@bot опиши" + reply на фото. Між цими світами лежить багато кроків:

```
Telegram update → adapter → unified_message → geometry → access check
              → user_task → planner → capability → response → render → send
```

Юніт-тест бере готовий `MessageGeometry(addressed_via_mention=True, ...)` і перевіряє один шматочок. Він **довіряє**, що `geometry` був правильно побудований. Якщо хтось зламає `build_geometry()` (парсинг mention у caption, у альбомі, з `via_bot`) — юніт-тест залишається зеленим, а бот ламається на проді.

### 1.2 Чому це повторюється

Моделі (включно зі мною), коли працюють над таском, бачать **локальну ціль** ("полагодь search-gate"). Вони не знають, які ще інваріанти покладаються на ту ж функцію, бо ці інваріанти **ніде не записані у виконуваній формі**. Документація розказує "як має бути", але не валиться при поломці.

> **Без тестового контракту кожна правка — це bullet, який просто пробиває стіну до своєї цілі. Решта будинку — щаслива випадковість.**

### 1.3 Що ми хочемо отримати

1. **Видимість стану.** В будь-який момент можна запустити одну команду і побачити: "31 інваріант працює, 5 нестабільні, 4 зламані". Зараз цього немає взагалі.
2. **Регресійний бар'єр.** Зламана поведінка не може непомітно потрапити в prod. Acceptance-тест валиться → deploy не йде.
3. **Спільна мова.** Користувач каже "бот не бачить фото в реплаї" — у документі є `B-016`, кожен (людина чи модель) одразу розуміє, що саме не працює.

## 2. Філософія: три рівні тестів

Smartest має **три** шари перевірок, кожен робить свою роботу. Не плутати між собою.

### 2.1 Юніт-тести (`tests/test_NNN_*.py`)

- **Що перевіряють:** окремі функції на ізольованих вхідних даних.
- **Стиль:** стандартний pytest. Можуть мокати все підряд.
- **Доля при рефакторі:** **можуть і повинні валитись**, якщо рефактор міняє внутрішню форму. Це нормально. Юніт-тести — не контракт, а сітка для розробки.
- **Приклад:** `test_061_message_logic_layers.py` — перевіряє, що `build_user_task` зі вхідним готовим `MessageGeometry` будує очікуваний `UserTask`. Не знає нічого про Telegram, mention-парсинг, групові чати.

### 2.2 Acceptance-тести (`tests/acceptance/test_*.py`)

- **Що перевіряють:** **продуктові інваріанти, сформульовані з точки зору користувача**, але через внутрішні рішення коду — не через текст відповіді LLM.
- **Стиль:** будуємо сирий Telegram update (як його бачить адаптер), ганяємо через **весь pipeline до точки рішення** (`build_geometry`, `build_user_task`, `plan_message`), читаємо **проміжний state**: значення полів `MessageGeometry`, `UserTask`, `ExecutionPlan`. LLM не викликаємо — стабаємо `chat_once` фіксованою відповіддю або взагалі обрізаємо до етапу planner.
- **Доля при рефакторі:** **святі**. Якщо рефактор їх ламає — або ти зламав поведінку, або контракт треба свідомо змінити (через окремий PR із оновленим `behavior-audit.md`).
- **Приклад:**
  ```python
  async def test_B016_mention_with_photo_reply_routes_to_vision():
      update = build_update(
          text="@botx опиши",
          reply_to=build_message(photo=True, file_id="abc"),
          bot_username="botx",
      )
      geometry = await build_message_geometry(unified_msg(update))
      task = await build_user_task(unified_msg(update), geometry, geometry.clean_text)
      plan = await plan_execution(123, task, geometry, authed_session())

      assert geometry.addressed_via_mention
      assert task.has_media_target
      assert task.media_type == "image"
      assert plan.capability in {"chat_vision", "chat_final"}
  ```

  Це **внутрішній** тест за формою (читає поля об'єктів), але **продуктовий** за змістом (формулюється "меншн із реплаєм на фото → бачить фото").

### 2.3 Smoke-тести (живий бот, ручні)

- **Що перевіряють:** end-to-end на staging-боті (`@chibigochibot`).
- **Стиль:** реальні повідомлення, реальні відповіді LLM.
- **Доля:** виконуються **раз перед prod-релізом**, коли acceptance-тести вже зелені.
- **Не входять у CI.** Бо нестабільні (LLM плавають), повільні (хвилини), коштують грошей.

> **Acceptance — це 95% страховки. Smoke — фінальна перевірка перед prod-cutover.**

## 3. Як писати acceptance-тест

### 3.1 Заборонено

- ❌ **Перевіряти текст відповіді моделі.** `assert "на фото" in response` — текст плаває, тест блимає, його зігнорують.
- ❌ **Викликати справжній LLM.** `monkeypatch.setattr(runner, "chat_once", fake_response)`.
- ❌ **Лізти у БД через ORM/SQL.** Юзай in-memory fakes або модельні fixture-и.
- ❌ **Тестувати "що буде якщо повна катастрофа".** Acceptance — про happy paths і явні error-paths з продуктового реєстру. Edge cases — справа юніт-тестів.

### 3.2 Обов'язково

- ✅ **Один тест = один інваріант з `behavior-audit.md`.** Назва тесту містить ID: `test_B016_*`.
- ✅ **Будуємо вхід як адаптер.** Сирий PTB Update (через builder-helpers у `tests/acceptance/conftest.py`).
- ✅ **Перевіряємо проміжні структури.** `geometry.X`, `task.Y`, `plan.Z`. Це і є "рішення коду".
- ✅ **Якщо інваріант про відповідь користувачу** — перевіряй, що **викликана відповідна capability** (`plan.capability == "tts"`), не текст.
- ✅ **При змінах spec-у — спочатку онови `behavior-audit.md`, тоді тест, тоді код.**

## 4. Three-state policy: 🟢 GREEN / 🟡 YELLOW / 🔴 RED

Реальність: коли ми починаємо аудит, **бот наполовину зламаний**. Не можна писати acceptance як "ось так має бути" і чекати, поки все полагодимо. Це pattern "напишемо тести коли все запрацює" → ніколи. Тому:

| Стан       | Значення                                              | Маркер у тесті                                            | Ефект                       |
|------------|-------------------------------------------------------|-----------------------------------------------------------|-----------------------------|
| 🟢 GREEN   | Працює зараз, користувач підтвердив                   | без `skip`/`xfail`                                        | Gate. Поломка → CI red.     |
| 🟡 YELLOW  | Працює частково / нестабільно                         | `@pytest.mark.xfail(reason="B-007: ...")`                 | Не блокує, але видно.       |
| 🔴 RED     | Зламано                                               | `@pytest.mark.skip(reason="B-012: not impl")`             | Не виконується, але видно.  |

**Принцип:** тест **виконуваний** (його тіло коректне) у всіх трьох станах. Різниця тільки в маркері. Як тільки 🔴 фікситься — знімаємо `skip`, тест стає gate-ом. Зворотно ламається — gate червоніє.

Один pytest-run видає табличку:
```
PASSED  31  (gate — must stay green)
XFAIL    5  (known fragile — track them)
SKIPPED  4  (known broken — fix in progress)
FAILED   0  (no regressions)
```

Це і є **поточний стан правди**. Кожен `SKIPPED` — це білет на полагодження. Кожен `XFAIL` — кандидат на стабілізацію.

## 5. Процес: як рухаємось від хаосу до контракту

### Ітерація 1. Інвентаризація (модель робить сама)

Модель проходить по коду (`adapters/telegram_bot.py`, `app/message_logic.py`, `agent/planner.py`, команди білінгу) і виписує **всі юзер-візибл функції, які бот ЗАЯВЛЯЄ підтримувати**. Не оцінює, не сортує — просто перелічує. Виходить `behavior-audit.md` v1: ~50 пунктів, кожен з:

- ID (`B-001`)
- Категорія (Commands / Addressing / Media / Search / Memory / Voice / Group / Billing)
- Тригер (що користувач робить)
- Гіпотеза (як я думаю воно має працювати)
- Поле "коментар користувача" (порожнє → гіпотеза правильна)
- Поле "статус" (порожнє → користувач не перевіряв ще)
- Code path (`file:line`)

### Ітерація 2. Тріаж із користувачем

Користувач проходить по списку. Для кожного пункту:
- Якщо гіпотеза правильна і функція працює → ставить 🟢
- Якщо гіпотеза правильна але функція блимає → ставить 🟡 + опціональний коментар
- Якщо гіпотеза правильна але функція зламана → ставить 🔴 + коментар про симптом
- Якщо гіпотеза НЕправильна → пише в коментарі, як насправді має бути
- Якщо такого взагалі не повинно бути → пише "видалити"

### Ітерація 3. Acceptance-код (модель робить сама)

Для кожного пункту з аудиту:
- 🟢 → тест без `skip`. Якщо падає на main — статус помилковий, обговорюємо.
- 🟡 → тест із `@pytest.mark.xfail(reason="B-NNN: <симптом>")`.
- 🔴 → тест із `@pytest.mark.skip(reason="B-NNN: <симптом>")`.
- "видалити" — викидаємо функцію з коду + не пишемо тест.

### Ітерація 4. Регресійний бар'єр

- `deploy/deploy.cjs` перед `tar czf` запускає `pytest tests/acceptance/ -x`.
- Не пройшло — деплою немає.
- Жодна модель не може це обійти, не виправивши або не довівши, що spec треба міняти (через зміну `behavior-audit.md` окремим PR).

### Ітерація 5. Поступове позеленіння

Сесія за сесією: беремо один 🔴 з `behavior-audit.md`, фіксимо, знімаємо `skip`, комітимо. Документ — robust ledger того, що залишилось.

## 6. Правила змін `behavior-audit.md`

Щоб документ не перетворився на сміття, дотримуємось:

1. **Додавання інваріанту** = чітке формулювання + acceptance-тест у тому ж комміті. Не можна додати пункт "колись напишу тест".
2. **Видалення інваріанту** = свідоме рішення користувача. Не можна "ой, цей пункт незручно полагодити, видалю".
3. **Зміна формулювання** = окремий PR, який користувач читає і свідомо мерджить. Це **зміна конституції**, не рефактор.
4. **Зміна статусу** (🔴 → 🟢) = тест зеленіє у поточному коді, або тест допасовано, що інакше нічого не змінилось.

---

## 7. Що це **не** робить

- ❌ Не замінює юніт-тести (вони продовжують жити для внутрішньої логіки).
- ❌ Не гарантує "бот точно працює правильно" — гарантує "бот працює у способах, які ми задокументували, не гірше".
- ❌ Не вирішує проблему якості відповідей моделі (це окрема історія: prompts, моделі, evaluation).
- ❌ Не охоплює UI/portal — там окремі smoke-тести (Playwright або хоча б ручні чек-листи).

## 8. Якщо ти модель і читаєш це

Перш ніж робити будь-яку зміну, що може торкнутись логіки бота:

1. **Прочитай `behavior-audit.md`.** Знайди пункти, які можуть бути зачеплені.
2. **Запусти `pytest tests/acceptance/ -v`.** Зафіксуй, що зараз зелене, що 🟡, що 🔴.
3. **Зроби зміну.**
4. **Запусти ще раз.** Якщо щось зеленого стало червоним — ти зламав інваріант. Зупинись, проаналізуй. Або відкоти, або онови `behavior-audit.md` свідомо (і дай користувачу шанс затвердити).
5. **Якщо твоя зміна повинна полагодити 🔴 → 🟢** — переконайся, що тест дійсно проходить, а потім зніми `skip`/`xfail`. Не сертифікуй себе сам — нехай pytest скаже "OK".

> **Без цього протоколу кожна правка — лотерея. З ним — інженерна робота.**

## 9. Як це виглядає на практиці (від bootstrap до steady state)

Це найважливіша секція для відповіді на питання: **"Я повинен 50 годин ручно проходити аудит?"** — Ні. Ось як це працює насправді.

### 9.1 Дві фази: Bootstrap і Steady state

```
ФАЗА 1: Bootstrap (один раз)              ФАЗА 2: Steady state (назавжди)
─────────────────────────────             ─────────────────────────────
Користувач проходить ~63 пункти    ──>    pytest tests/acceptance/ (4 сек)
руками в Telegram                          автоматом перед кожним deploy
        ↓                                            ↓
Модель пише acceptance-тести                Зелено → deploy. Червоно → стоп.
для кожного пункту з відповідним                    ↓
маркером (🟢=без skip, 🟡=xfail, 🔴=skip)   Користувач не торкається аудиту,
        ↓                                  поки не з'являється нова продуктова
Зеленіє baseline. CI gate активний.        фіча.
```

**Bootstrap робиться ОДИН РАЗ.** Не повторюється при кожній зміні коду.

### 9.2 Скільки виконується pytest acceptance

`pytest tests/acceptance/` — це **не** запуск живого бота. Це чистий Python:

- Будуємо fake Telegram update (через builder-helpers)
- Ганяємо через `build_geometry()` → `build_user_task()` → `plan_message()`
- Перевіряємо поля проміжних структур (`MessageGeometry`, `UserTask`, `ExecutionPlan`)
- LLM замокано фіксованою відповіддю (`monkeypatch.setattr(runner, "chat_once", fake_response)`)

Один тест — **30-150ms**. Усі ~63 — **3-5 секунд**. Ти не сидиш годину; ти бачиш `63 passed in 4.2s`.

### 9.3 Сценарій: Codex додає нову фічу (наприклад, `/weather Київ`)

**Крок 1.** Codex пише код (новий handler, route, capability).

**Крок 2.** Перед `git push` (або в pre-commit, або в `deploy.cjs`) автоматично запускається:
```bash
pytest tests/acceptance/ -v
```

**Крок 3a — нічого не зламано:**
```
63 passed in 4.2s
```
Codex додав погоду, не зачепив `/c@saintaibot`, не зламав vision, не зламав search-gate. Деплой проходить.

**Крок 3b — Codex випадково зламав інваріант:**
```
61 passed, 2 failed
FAILED tests/acceptance/test_addressing.py::test_B001_clear_command_requires_botname
FAILED tests/acceptance/test_media.py::test_B016_reply_on_photo_with_mention_routes_to_vision
```
Деплой блокується. Користувач бачить **які саме** інваріанти зламані. Каже Codex "ти зламав B-001 і B-016, виправ" — Codex фіксить. Без gate-у це знайшлось би через тиждень — від користувача в Telegram-чаті.

### 9.4 Коли користувач знов відкриває `behavior-audit.md`

Тільки в трьох випадках:

| Випадок                                          | Що робить користувач                              | Скільки часу |
|--------------------------------------------------|---------------------------------------------------|--------------|
| Нова фіча (модуль погоди)                        | Додає 1-3 нові пункти `B-064 ... B-066`. **Не торкається** існуючих 63. Каже моделі "напиши тести під ці нові пункти". | 5 хвилин     |
| Зміна існуючого інваріанту (`/c` без тегу теж очищає) | Окремий PR, переписує `B-001`, модель підлаштовує тест. Решта не чіпається.                          | 10 хвилин    |
| Раз на квартал — санітарна перевірка             | Проходить 5 ключових пунктів руками, щоб переконатись що моки не розійшлись з реальністю.            | 15 хвилин    |

50 нових фіч = **50 нових пунктів аудиту** (по 1-3 на фічу), **не повторне проходження старих 63**. Кожна фіча додає тести, старі продовжують працювати автоматично.

### 9.5 Хто і коли запускає acceptance

| Роль                  | Коли                                  | Команда                       |
|-----------------------|---------------------------------------|-------------------------------|
| Модель (Claude/Codex) | Перед кожним push після зміни логіки  | `pytest tests/acceptance/`    |
| `deploy.cjs`          | Автоматично перед `tar czf`           | вбудовано в скрипт            |
| Користувач            | Ніколи (хіба перевірити "а що там")   | те саме                       |
| CI (якщо буде)        | На кожен PR                           | у GitHub Actions / аналог     |

### 9.6 Порівняння з альтернативами

| Підхід                          | Швидкість      | Покриття       | Ламає Codex регулярно? |
|---------------------------------|----------------|----------------|------------------------|
| Тільки юніт-тести (330)         | Швидко (5s)    | Внутрішня логіка| **Так** — продукт незахищений |
| Тільки smoke (живий бот)        | Повільно (год+)| Все, кінцеве   | Ні, але повільно ловиш |
| **Acceptance + smoke перед prod** | **Швидко (4s)**| **Продуктові інваріанти** | **Ні** — fast feedback |
| 50 годин ручної роботи          | 50 годин/раз   | Все             | Не масштабується       |

### 9.7 Цикл життя одного інваріанту

```
[ручний прохід один раз] → 🟢/🟡/🔴 у audit → acceptance-тест → CI gate
                                                                      ↓
[Codex робить зміну] → pytest за 4 секунди → fail/pass → дія
                                                                      ↓
[Codex чи людина] фіксить регресію або підтверджує "так, я свідомо змінив"
                                                                      ↓
[якщо свідома зміна] → окремий PR на behavior-audit.md → нова версія тесту
```

Користувач у цьому циклі — **тільки** на кінцях:
- На початку (bootstrap-прохід один раз)
- Коли свідомо змінює правило (PR на audit)
- Коли Codex просить "я зламав B-016, чи це OK?" — користувач каже так чи ні

**Не**: користувач НЕ повторює 63-пунктний прохід при кожній зміні коду. Це й була основна суть.
