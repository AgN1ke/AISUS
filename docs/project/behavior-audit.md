> ⚠️ Тут НЕМАЄ команд білінгу (`/balance`, `/topup`, `/settings`, `/model`), owner-команд (`/mode`, `/allow`, `/ban`, `/cap`), порталу і паролів. Multi-tenant фічі — це окремий проєкт `multitenant`-гілки і `@chibigochibot`. Не плутати.

# Прийняті рішення (Session 101 round 2)

Після першого проходу зафіксовано:

1. **Reasoning тригери** — тільки `/think`. 🧠 emoji (B-004) і фрази "use reasoning" (B-005) — DROP, бо ненадійні.
2. **Reasoning видимість** — додаємо debug-маркер `🧠 [reasoning ON]` в кінці відповіді коли `/think` спрацював. Інакше неможливо перевірити з боку користувача (новий пункт **B-069**).
3. **Цитати в пошуку** — рефактор на domain-based формат `[google.com](url)` / `[nasa.gov](url)`. Це простіше і людяніше за плутану нумерацію `[1]` `[5]` `[1]` (новий пункт **B-070**).
4. **`/v` як reply** — DROP (B-012). Це функціонал `/a`, не дублюємо.
5. **B-031 reply-ланцюжки** — переформульовано: бачить тільки прямий reply-target (1 hop), бо Telegram забороняє ботам читати чужі повідомлення.
6. **B-056 контекст** — переформульовано: весь recent + long + CORE, не "тільки 6 повідомлень".

## A. Команди

### B-001 — `/c@saintaibot` очищає всю пам'ять цього чату

- **Як перевірити:** напиши `/c@saintaibot` у будь-якому чаті (приваті чи групі).
- **Що має статись:** бот стирає **всі шари пам'яті**: recent (поточна переписка) + long-term (стиснуті спогади) + CORE-факти (профіль учасників) + podcast_pending / dossier (якщо були). Підтверджує: "Контекст цього чату повністю очищено".
- **Що НЕ має статись:** bare `/c` (без тегу) **не повинен очищати**. Форма `@saintaibot /c` (тег спереду, команда ззаду) теж **не повинна очищати**. Тільки строго `/c@saintaibot`. Це принципово, бо в груповому чаті можуть жити кілька ботів, і bare `/c` не повинен чіпати пам'ять жодного.
- **Статус:**
- **Коментар:** хз, чи він чистить, але повідомлення видає

- **Що НЕ має статись:** не повинен видавати фейкову історію розмови ("ти питав про X, я відповідав про Y").

## I. Інфраструктурні правила (рідко видно прямо)

### B-054 — Stale media файли чистяться при старті

- **Як перевірити:** опосередковано — якщо бот падав/перезавантажувався під час обробки медіа, при наступному старті цей tmp-файл не повинен заважати.
- **Що має статись:** на startup `purge_stale_media_tmp()` чистить старі media tmp-файли.
- **Що НЕ має статись:** старі file_id з попередньої сесії не повинні використовуватись (Telegram їх інвалідує).

### B-056 — Контекст: весь recent + long + CORE, фокус на останньому

- **Як перевірити:** опосередковано — бот при відповіді не "забуває" що було 20 повідомлень тому в межах поточного recent-бюджету.

# Що далі

Коли пройдеш — я (модель) читаю заповнений документ і:

1. Для 🟢 — пишу acceptance-тест без skip. Якщо тест на main валиться — повертаюсь до тебе уточнити.
2. Для 🟡 — тест із `xfail` ("очікувана нестабільність").
3. Для 🔴 — тест із `skip` + reason з твого коментаря. Список боргу.
4. Для 🗑 — окремий PR на видалення коду.
5. Інтегрую `pytest tests/acceptance/` у `deploy.cjs` як обов'язковий gate перед заливкою на prod.
6. Сесія за сесією — беремо один 🔴, фіксимо, переводимо в 🟢.
